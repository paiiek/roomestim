| **Item I — Sample rate** | **48000 Hz** throughout the pipeline | Matches spatial_engine default (`spatial_engine/configs/lab_8ch.yaml` rate=48000). Reverse: drop to 16000 Hz only if HF Spaces CPU envelope cannot sustain 48 kHz ISM at `max_order=10` on a 6-room-corner geometry. |
| **Item J — ISM max_order** | **10** (default); fallback **6** if §0.4 STOP rule #4 fires | Vorländer (2020) §11 recommends `max_order ≥ 8` for RT60 ≤ 1 s rooms (typical lab/conference); 10 gives a safety margin without doubling compute cost (image-source count is O(N^3); 10^3 = 1000 vs 8^3 = 512). HF Spaces free-tier CPU envelope expected to render 30 s output in ~5-15 s at order 10 for a 6-corner shoebox. Reverse: drop to 6 if render exceeds 30 s wall time (perceived as broken UX). |
| **Item K — FFT size for convolution** | **8192** samples (block-wise overlap-add via `scipy.signal.fftconvolve` on per-band IR chunks; output length = `len(source) + len(ir) - 1` with overlap-add for efficiency) | Standard partitioned-convolution block size for 48 kHz audio at the IR lengths expected (typical RT60 = 0.6 s → IR ≈ 28800 samples; FFT 8192 gives 7 overlapping blocks per IR). Reverse: if memory ceiling fires on HF Spaces (4 GB RAM free tier), drop FFT size to 4096. |
| **Item L — Speaker → ISM SoundSource mapping** | One `pra.SoundSource` per `PlacedSpeaker` in `layout.yaml`, at the speaker's `(x, y, z)` listener-frame position translated to pyroomacoustics' world-frame (origin at `floor_polygon[0]`); signal = the loaded source clip (mono → broadcast to all 8 sources) | The demo is "what would the source sound like in the user's room with the 8 placed speakers playing it"; broadcasting the same source across all 8 speakers is the simplest faithful render. A future v0.12-web.1 may add per-channel signal differentiation (e.g., VBAP-encoded virtual source path), but v0.12-web.0 keeps the demo simple. Reverse: switch to a single-source render at the listener centroid if 8-source ISM proves too slow. |
| **Item M — Mic / listener position** | One `pra.MicrophoneArray` with 2 mic positions at `(listener_centroid_x ± 0.0875, listener_centroid_y + 0.0, listener_centroid_z + 0.0)` (HUTUBS measured 17.5 cm inter-aural distance) | Binaural rendering requires a 2-channel mic array; HUTUBS inter-aural distance 17.5 cm is standard. Each mic's per-arrival signal gets convolved with the corresponding HRIR (azimuth + elevation derived from arrival-direction-of-arrival per image source). Reverse: if the per-image-source DOA computation proves intractable, drop to a single mic at the centroid + post-hoc binaural-from-mono using HUTUBS direct-path HRTF only (worse but renderable). |
| **Item N — HRTF binaural rendering algorithm** | **per-image-source DOA → HRTF lookup → time-aligned convolution → sum**: for each image source from pyroomacoustics, compute the DOA (azimuth + elevation) in the listener's head-relative frame; look up the closest HUTUBS HRIR by Euclidean angular distance; convolve the per-source signal with the HRIR; time-align by per-source arrival delay (`distance / speed_of_sound`); sum across image sources into final binaural stereo. | This is the standard "binaural ISM" approach (cf. Schissler & Manocha 2016, Stitt et al. 2018). Alternative: pyroomacoustics' built-in `Room.simulate()` + post-hoc binaural convolution of the mono output. Rejected: the post-hoc approach loses DOA per image source (everything is binauralised from the listener-centroid mono signal — physically incorrect for the early reflections that are the perceptual signature of room acoustics). Reverse: only if the per-image-source path proves too slow (> 30 s) AND a v0.12-web.1 critic verdict argues the perceptual difference is below JND. |
| **Item O — Output normalization** | Peak-normalize to **-1 dBFS** (max abs sample = 0.891) post-convolution; no compression / EQ | Standard demo-WAV normalization. The -1 dBFS ceiling avoids clipping when played back through users' systems (which may add headroom or master limiting). Reverse: switch to LUFS-normalisation (target -16 LUFS, broadcast spec) if the demo proves perceptually quieter than user expectation. |
| **Item P — Output WAV format** | **48 kHz, stereo, 16-bit PCM RIFF/WAV** (via `soundfile.write(path, audio, 48000, subtype='PCM_16')`) | 16-bit is browser-compatible (HTML5 `<audio>` does not need 24-bit/32-bit float); 48 kHz preserves the simulation sample rate. Reverse: switch to 24-bit if the -1 dBFS peak normalisation reveals dither artefacts (unlikely at 30 s demo length). |

### §0.1 Why-this-scope-not-alternatives (DELIBERATE-mode discipline)

| Alternative considered | Status | Reason |
| --- | --- | --- |
| Bundle web demo into `roomestim/` core package (no sibling) | Rejected | Pollutes core import surface with Gradio/Plotly/pyroomacoustics heavy deps; breaks the v0.11 §0.3 invariant that core stays minimal. |
| Defer to v0.13-web or later | Rejected | Parent-prompt explicit: ship at v0.12-web. The acoustics-track v0.13 has already shipped; deferring further does not free engineering capacity. |
| Skip binaural rendering (5 views → 4) | Rejected | Parent-prompt calls binaural the "killer feature"; dropping it gut-punches the demo's value proposition. |
| Skip setup PDF (5 views → 4) | Rejected | Parent-prompt explicit; PDF is the "take to the actual room with a tape measure" deliverable. |
| Skip 3D viewer (5 views → 4) | Rejected | Parent-prompt explicit; Plotly mesh is the visual hook for the demo. |
| Download HRTF on first use vs. bundle | Rejected (download) | HF Spaces cold-start should not block on network; HUTUBS URL has rotated; ~1 MB bundle is negligible. |
| Use OpenAIR HRTF instead of HUTUBS | Rejected | OpenAIR is HRIRs for rooms (already-measured RIRs), not anatomy-measured head HRTFs; wrong shape for this pipeline. |
| Use SADIE-II HRTF instead of HUTUBS | Possible | SADIE-II is CC BY 4.0 + York; comparable to HUTUBS. HUTUBS chosen for parent-prompt-explicit lock. Reverse criterion in §10. |
| Use Streamlit instead of Gradio | Rejected | Gradio is HF-Spaces-canonical; cleaner Plotly + Audio integration. |
| Skip ZIP download (5 views → 4 + individual files) | Rejected | Single-ZIP is the "save the whole result" convenience; aids reproducibility (provenance README inside ZIP). |

## §4 Pre-mortem (3 failure scenarios; DELIBERATE-mode requirement)

### Scenario S1 — binaural render exceeds HF Spaces CPU envelope (> 30 s wall time)

**Failure mode**: ISM at `max_order=10` with 8 sound sources + per-image-source HRTF convolution on 30 s of source audio at 48 kHz exceeds the perceived-broken UX threshold. Users abandon the demo at the loading spinner.

**Mitigation order**:
1. Drop `max_order` to 6 (§0.0 Item J reverse).
2. Drop sample rate to 24 kHz (§0.0 Item I reverse; HUTUBS resampled accordingly).
3. Pre-render the binaural output once per `(algorithm, n_speakers)` combination and cache to disk; serve from cache when sidebar matches.
4. If all three fail: drop binaural to "single direct-path HRTF + room IR convolution" (no per-image-source DOA); document under v0.12-web.1 successor ADR.

**Detection at design time**: P13e §5 unit test measures wall time; if median exceeds 20 s on local dev machine, escalate before ship.

### Scenario S2 — HUTUBS bundle SHA-256 changes between download instances

**Failure mode**: the bundled `hutubs_pp1.sofa` differs byte-equal from a reference SHA-256 because TU Berlin re-issues the file (corrects a metadata field, fixes a sample index). Golden-byte-exact binaural regression test (§24 #ii) flakes.

**Mitigation**:
1. Pin the SHA-256 of the bundled SOFA file in `roomestim_web/data/hrtf/HRTF_ATTRIBUTION.md`.
2. CI gate verifies `sha256sum roomestim_web/data/hrtf/hutubs_pp1.sofa` against the pinned hash; fails fast if mismatch.
3. If TU Berlin re-issues, update the pinned hash + re-record the binaural-golden SHA-256 + bump `roomestim_web/__version__` to `0.12-web.1`.

**Detection at design time**: SHA-256 pin captured at P13e commit time.

### Scenario S3 — Gradio 4.x API drift breaks `app.py` on HF Spaces

**Failure mode**: HF Spaces installs the latest Gradio version which has deprecated an API we depend on (`gr.Plot` → `gr.Plotly` rename, or `gr.File` upload behaviour change). Spaces build fails or app crashes on first request.

**Mitigation**:
1. Pin `gradio==4.X.Y` in `requirements.txt` (not `>=`).
2. Pin `plotly==5.A.B`, `pyroomacoustics==0.7.X`, `pysofaconventions==X.Y`, `reportlab==4.X`, `soundfile==0.Z.W`.
3. CI gate (future v0.12-web.1+) runs the web tests in a clean Python 3.11 env matching HF Spaces' default.
4. If the pin diverges from a security-patched version (CVE in pinned Gradio), bump pin + re-test before deploy.

**Detection at design time**: P13g checks `requirements.txt` against HF Spaces' actual install at deploy time; user runs `gradio` locally with the pinned version before pushing.

```
    room: RoomModel,                # from roomestim.model
    layout: PlacementResult,        # from roomestim.model
    source_audio_path: Path,        # bundled at roomestim_web/data/audio/source.wav
    hrtf_subject: str = "pp1",      # HUTUBS subject id
    sample_rate: int = 48000,
    max_order: int = 10,
    out_path: Path,
) -> None
```

### §5.2 Step-by-step algorithm

1. **Load source audio**: `audio_mono, sr = soundfile.read(source_audio_path)`; assert `sr == 48000`; assert `audio_mono.ndim == 1`; assert `28.0 <= len(audio_mono) / sr <= 32.0`.

2. **Build pyroomacoustics room**:
   - Convert `room.floor_polygon` (list[Point2]) into a 2D numpy array `corners = np.array([[p.x, p.z] for p in room.floor_polygon]).T`.
   - For each `room.surfaces[i]`, build a `pra.Material` from `surface.absorption_bands` if present (per-band absorption dict `{"absorption": {"a125": ..., "a250": ..., ..., "a4000": ...}}`) else from `surface.absorption_500hz` (scalar). pyroomacoustics accepts `pra.Material(absorption_coefficient_or_bands)` either way.
   - `room_pra = pra.Room.from_corners(corners, fs=sample_rate, max_order=max_order, materials={surface_idx: material}, ray_tracing=False, air_absorption=True)`.
   - `room_pra.extrude(room.ceiling_height_m, materials=...)` to make 3D shoebox-like extrusion.

3. **Place sources** (per §0.0 Item L):
   - For each `placed_speaker` in `layout.speakers`:
     - `src_pos = np.array([placed_speaker.position.x, placed_speaker.position.z, placed_speaker.position.y])` (note: pyroomacoustics uses (x, y, z) where z is vertical; our convention has y vertical and z front).
     - `room_pra.add_source(src_pos, signal=audio_mono, delay=0.0)`.

4. **Place listener microphones** (per §0.0 Item M):
   - `centroid = room.listener_area.centroid; height = room.listener_area.height_m`.
   - `mic_left_pos = np.array([centroid.x - 0.0875, centroid.z, height])`.
   - `mic_right_pos = np.array([centroid.x + 0.0875, centroid.z, height])`.
   - `mic_array = pra.MicrophoneArray(np.array([mic_left_pos, mic_right_pos]).T, fs=sample_rate)`.
   - `room_pra.add_microphone_array(mic_array)`.

5. **Run ISM**: `room_pra.image_source_model()` (computes image sources; deterministic given fixed inputs).

6. **Compute per-image-source DOA + HRTF convolution** (the core of §0.0 Item N; replaces the default `room_pra.simulate()` which produces non-binaural mono-per-mic output):
   - Load HRTF table: `hrtf_table = load_hutubs(hrtf_subject)`.
   - For each source `s` in `room_pra.sources`:
     - Get image-source positions and their respective gains/delays: `images = s.images` (shape `(3, N_images)`), `damping = s.damping`, `orders = s.orders`.
     - Initialize per-source binaural output buffer `out_stereo = np.zeros((2, len(audio_mono) + max_ir_len + max_delay_samples))`.
     - For each image `i` in range `N_images`:
       - Direction-of-arrival from image to listener-centroid: `doa = (images[:, i] - centroid_3d) / np.linalg.norm(...)`.
       - Convert DOA to (azimuth, elevation) in head-relative frame.
       - Look up HRIR pair: `hrir_l, hrir_r = hrtf_table.nearest_neighbor(az, el)`.
       - Compute delay in samples: `dly = int(np.round(distance_to_image / speed_of_sound * sample_rate))`.
       - Compute amplitude gain from `damping[i]` (per-band attenuation; for v0.12-web.0 collapse to broadband by taking the mean across the 6 bands).
       - Convolve source signal × gain with HRIR: `conv_l = scipy.signal.fftconvolve(audio_mono * gain, hrir_l, mode='full')`; similarly right.
       - Time-align by `dly` and accumulate into `out_stereo[0, dly:dly+len(conv_l)] += conv_l`; right channel similarly.
   - Sum `out_stereo` across all sources into a final stereo array.

7. **Normalize + write**:
   - Trim trailing silence beyond `len(audio_mono) + 2 * sample_rate` samples (2 s tail max).
   - Peak-normalize to -1 dBFS: `peak = np.max(np.abs(out_stereo)); out_stereo *= (10 ** (-1.0 / 20.0)) / peak`.
   - `soundfile.write(out_path, out_stereo.T, sample_rate, subtype='PCM_16')`.

### §5.3 Determinism

- `numpy.random.seed(0)` before `room_pra.image_source_model()`.
- pyroomacoustics ISM is deterministic given fixed inputs (no random ray tracing in our path; `ray_tracing=False`).
- HRTF nearest-neighbor lookup is deterministic (Euclidean angular distance; tie-break by lower index).
- `scipy.signal.fftconvolve` is deterministic.
- → byte-exact output WAV bytes given fixed `(room.yaml, layout.yaml, source.wav, hutubs_pp1.sofa, max_order, sample_rate)`.

### §5.4 Octave-band material mapping

For each `Surface`:
- If `surface.absorption_bands` is set (the room.yaml octave-band block was populated), pass `pra.Material(energy_absorption={"description": "from room.yaml", "coeffs": [a125, a250, a500, a1000, a2000, a4000], "center_freqs": [125, 250, 500, 1000, 2000, 4000]})`.
- Else fall back to scalar: `pra.Material(energy_absorption=surface.absorption_500hz)`.
- pyroomacoustics' internal band-pass filter bank handles the band convolution.

## §13 ADR stubs to be written before executor starts

### ADR 0024 — Web demo as separate package (`roomestim_web/`)

- **Status (to be written)**: Accepted (v0.12-web.0).
- **Decision**: Web demo lives in a sibling top-level package `roomestim_web/` (NOT inside `roomestim/`). Gated behind `pip install -e ".[web]"` optional extra. Core `roomestim/` import surface and dependencies BYTE-EQUAL except for the `roomestim/place/dispatch.py` placement refactor that exposes a public API. Cross-ref: D29, D30, parent design plan §0.0 Item A.
- **Drivers**: (i) Demo-only heavy deps (Gradio, Plotly, pyroomacoustics, HRTF data) must not pollute core install; (ii) HF Spaces deploy requires a clean top-level `app.py`; (iii) v0.11 §0.3 invariant on core stability.
- **Alternatives considered**: (a) Bundle in core — rejected (pollutes core). (b) Separate git repo `roomestim-web` — rejected (decouples release cadence too far; cross-repo bootstrapping pain). (c) Subpackage `roomestim/web/` — rejected (still pollutes core import surface).
- **Consequences**: (+) Core stays minimal. (−) Two packages to maintain. (−) Refactor of `cli.py::_run_placement` → `place/dispatch.py` is required (P13a).
- **Reverse-criterion**: Collapse to a single package if web demo becomes mandatory (e.g., the project pivots to "the web demo IS the product").

### ADR 0025 — Binaural demo stack (pyroomacoustics ISM + HUTUBS HRTF + per-image-source DOA convolution)

- **Status**: Accepted (v0.12-web.0).
- **Decision**: Render binaural demo via the `roomestim_web/binaural.py` pipeline §5.2 §5.3. Stack: pyroomacoustics ≥ 0.7 (ISM, image_source_model + air_absorption=True, ray_tracing=False, max_order=10, fs=48000) + pysofaconventions HRTF loader + scipy.signal.fftconvolve. Algorithm: per-image-source DOA → HUTUBS nearest-neighbor HRIR → time-aligned convolution → sum.
- **Drivers**: (i) Parent-prompt locks the stack; (ii) per-image-source DOA preserves early-reflection coloration (the perceptual signature of room acoustics); (iii) pyroomacoustics ISM is deterministic given fixed seed.
- **Alternatives considered**: (a) Post-hoc binaural-from-mono — rejected (loses early-reflection DOA). (b) Ray tracing — rejected (non-deterministic at small ray counts; out of pyroomacoustics ISM lock). (c) Wave-based simulator — rejected (out of scope).
- **Consequences**: (+) Deterministic; (+) byte-exact golden test feasible. (−) max_order=10 may exceed HF Spaces CPU envelope (§4 S1).
- **Reverse-criterion**: Drop max_order to 6 or sample_rate to 24 kHz on S1 firing.

### ADR 0026 — HRTF dataset selection (HUTUBS pp1 PRIMARY; KEMAR FALLBACK)

- **Status**: Accepted (v0.12-web.0).
- **Decision**: HUTUBS subject `pp1` (TU Berlin; CC BY 4.0) PRIMARY HRTF; MIT KEMAR (Public Domain) FALLBACK. Both bundled in-repo at `roomestim_web/data/hrtf/`. Resampled 44.1 → 48 kHz at load time.
- **Drivers**: (i) Parent-prompt locks HUTUBS + KEMAR; (ii) bundling avoids HF Spaces cold-start network dependency; (iii) ~1 MB combined bundle is negligible vs the 10 MB §0.4 STOP threshold.
- **Alternatives considered**: (a) SADIE-II — possible alternative (also CC BY 4.0); rejected for parent-prompt explicit lock. (b) Download-on-first-use — rejected (cold-start UX + URL rotation risk).
- **Consequences**: (+) Deterministic; (+) attribution-only commercial-safe; (−) one-subject pp1 may not personalise for all users (OQ-17).
- **Reverse-criterion**: Switch PRIMARY to SADIE-II if HUTUBS license terms change.
