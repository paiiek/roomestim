## TL;DR — 한 문장

**v0.8.0까지 8개 릴리스를 찍었지만, 원래 v0.1 acceptance gate(A10 lab capture)는 단 한 번도 통과되지 않았다.** v0.3 ~ v0.8 (6개 릴리스 / 약 70%의 작업)는 ACE Challenge라는 *외부 연구 데이터셋*의 RT60 예측 품질에 투입되었고, 정작 프로젝트가 풀려고 했던 "사용자의 실제 랩 룸을 phone-scan → 엔진 통합"은 v0.1.1 closeout 이후 진척이 0이다.

## 1. 원래 비전 (v0 design, 2026-05-03)

### 1.1 North star
> Phone room-scan (RoomPlan / Polycam / COLMAP fallback) → simplified `RoomModel` + algorithm-aware speaker placement → engine-ready `layout.yaml` + new `room.yaml` schema. **Cm-grade installation work**: walls ±10 cm, speaker angles ±2-5°, RT60 ±20%.

### 1.2 v0.1 ship 조건 (Q9 / A10)
1. `roomestim ingest --backend roomplan tests/fixtures/lab_room.usdz` → corner err <10 cm vs hand-tape GT
2. `roomestim place --algorithm vbap --n-speakers 8` → `layout.yaml`이 spatial_engine 로더 통과
3. `roomestim export` → 두 YAML 파일 작성, `room.yaml`은 자체 schema 검증
4. CI green

### 1.3 진짜 acceptance gate (A10)
> 수동 캡처 `tests/fixtures/lab_real.usdz` (실제 랩룸 RoomPlan scan)이 (a) corner-vs-tape err <10 cm, (b) VBAP-8 layout이 ±5° azimuth + ±10 cm radial로 실제 배치된 스피커와 일치.

**A10이 v0.1 release의 게이트.** 즉, A10 통과 전까지 프로젝트는 *공식적으로 v0.1을 완료하지 못한* 상태다.

## 4. Stage-2 schema lock 상태

D2 / D8 / OQ-4에 따르면 Stage-2 schema lock (`__schema_version__ = "0.1"`, `additionalProperties: false`)은 **A10 통과를 전제 조건으로** 한다.

현재:
```python
__schema_version__ = "0.1-draft"  # Stage 1 permissive — v0.8.0까지 unchanged
```

→ **v0.1 schema는 8개 릴리스가 지나도록 lock-in되지 못했다.**
→ 이는 v0.2의 cross-repo PR이 draft에서 멈춘 이유이기도 하다 (schema가 unfrozen이면 spatial_engine이 받을 수 없음).

## 6. 드리프트 분석 — 왜 ACE 작업으로 빠졌나

v0.3.0 시점에 결정이 갈렸다:
- **Path A**: A10 게이트 = 사용자 실제 랩룸을 직접 캡처하고 결과 검증 → 물리적 액세스 + 시간 필요 (외부 의존)
- **Path B**: ACE Challenge 같은 공개 데이터셋으로 RT60 predictor를 *연구 수준으로* 검증 → 코드만으로 진행 가능 (자기충족적)

선택된 길: **Path B**. 이유는 합리적이다 — 랩 캡처 세션이 가능하지 않은 시점에 "할 수 있는 것"을 했고, A11 (RT60 ±20%)을 우회적으로 강화하려고 했다. ADR discipline + 솔직한 caveat 표시는 모두 잘 됐다.

문제: **Path B가 6번 릴리스 동안 self-reinforcing loop**가 됐다. v0.7 critic이 정확히 짚었듯이:
- 매 릴리스마다 deferred 항목 리스트가 늘어나기만 하고 줄지 않는다 (3 → 8 → 8 → 10)
- 매 릴리스가 또 다른 ADR을 만들어내는 것이 자기 정당화의 패턴이 된다
- 진짜 게이트(A10)는 전체 작업 흐름 밖에 있다

v0.8 null result는 이 패턴에서 약간 벗어났다 — 단일 계수 가설을 *기각*해서 deferred 리스트를 한 항목 줄였다. 그러나 여전히 ACE corpus 차원에서 행해진 일이고, 사용자의 실제 랩룸과는 무관하다.

## 8. v0.8.0 자체 평가

긍정:
- v0.7 critic의 4개 finding (C1, M1, M2, M3) 모두 직접 처리
- Bracketing experiment는 **falsifiable null result**를 만들어냄 — F3 + lecture_seat 단독 가설을 기각함으로써 deferred 리스트를 한 항목 줄임
- Per-band MAE snapshot test는 future PR이 residual coverage를 자동 평가받게 만들어준 진짜 인프라
- Override mechanism이 깨끗 (frozen dataclass, no module-level mutation)
- OMC orchestration (planner → executor → reviewer → verifier) 풀 사이클로 작동, 22개 hard exit criteria 모두 검증

부정:
- 여전히 ACE corpus 차원의 작업 — 사용자 랩룸과 무관
- "null verdict"는 honest하지만, 동시에 "F3 단독으로 안 되니 v0.9에서 더 큰 sweep + coupled-space 모델"로 *또 한 번 deferred*되는 패턴
- v0.8 새 fixture (`ace_eaton_2016_table_i_measured_rt60.csv`)는 v0.6 perf doc을 옮겨 적은 것 — 새로운 측정 데이터가 아님

## 9. 진짜 다음 스텝 — 3가지 전략 옵션

ACE corpus loop를 계속하는 것은 v0.9 / v0.10 / v0.11에서도 가능하다. 그러나 그것은 north star로부터 더 멀어지는 길이다. 솔직하게 3가지 옵션:

### Option α — A10 lab capture를 *진짜로* 잡고 v0.1 게이트 닫기
**Acceptance**: `lab_real.usdz` 캡처 (iPad Pro RoomPlan 또는 Polycam) + tape-measured corner GT 3개 + 실제 8-channel VBAP 배치 + 측정. 게이트 통과 시 Stage-2 schema flip + cross-repo PR 머지.
**비용**: 물리적 액세스 + 약 1일 (캡처 + 측정 + 검증)
**효용**: v0.1 게이트 닫힘 → v0.2 cross-repo PR이 unblock → spatial_engine submodule attach 결정 가능
**리스크**: 캡처 결과가 ±10 cm를 못 맞추면 reconstruction에 실제 작업 들어감 (그러나 그것이 진짜 정보)
**추천**: ★★★ 원래 비전을 closeout하는 유일한 길

### Option β — v0.9 F4a sweep + coupled-space predictor (ACE 계속)
**Acceptance**: `MaterialAbsorptionBands[wall_painted]` + `[ceiling_drywall]` 2k/4k에서 ±25% 범위 내 sensitivity sweep; OR Cremer/Müller two-room formula scaffold (Lecture_2 multi-slope decay 가설)
**비용**: 코드만 (대략 v0.6/v0.7 수준)
**효용**: ACE corpus residual을 더 줄임; v0.8 null result에서 elevated된 가설 검증
**리스크**: critic이 다시 "ADR theatre" 카드 꺼낼 가능성; deferred 리스트는 또 늘어남; 사용자 랩과 여전히 무관
**추천**: ★ (사용자가 명시적으로 ACE 더 파고 싶다면)

### Option γ — Spatial_engine 실제 binary로 A2 / A15 unblock
**Acceptance**: `SPATIAL_ENGINE_BUILD_DIR` 환경 변수 세팅 + `layout_loader_smoke` C++ 바이너리 빌드 + parity test 1000 hypothesis triples 통과. A2 / A15가 SKIP에서 PASS로.
**비용**: spatial_engine 빌드 환경 구성 (CMake, 의존성). 사용자가 이 빌드를 이미 한 적 있는지 확인 필요
**효용**: Cross-repo 통합이 시방서 수준이 아닌 *실제 binary 수준*에서 검증됨
**리스크**: spatial_engine 빌드가 깨졌을 수 있음 (8개월간 미통합); 빌드 자체가 별도 작업 됨
**추천**: ★★ Option α의 prerequisite로 자연스럽게 따라옴

### 권장 순서
1. **Option α를 먼저** — A10 캡처 세션을 일정에 넣는다. 이걸 안 하면 v0.1은 영원히 닫히지 않는다.
2. **Option γ를 동시 또는 직후** — 캡처 결과 검증 시 실제 엔진 binary roundtrip이 의미 있다.
3. **Option β는 A10 통과 *후에*** — 진짜 랩 데이터에서 RT60 ±20% 검증 후, 그래도 부족하면 그때 더 깊은 RT60 작업.

## 10. 전체 한 줄 요약

> roomestim v0.8.0은 정밀한 RT60 연구 도구가 됐다. 그러나 그것이 **원래 만들고자 했던 도구는 아니다.** 원래 도구 — 사용자의 실제 랩 / 전시 공간을 phone-scan해서 cm-grade 스피커 배치를 산출하고 spatial_engine에 먹이는 capture-to-config 도구 — 은 v0.1.1 이후 8개월간 진척 0이다. v0.9의 가장 정직한 첫 동작은 *코드를 한 줄도 쓰지 않는 것*: 일정에 1일을 잡아 lab capture 세션을 진행하는 것.
