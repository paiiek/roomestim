## 1. Ground truth — what the code actually does (verified at `2ad9ece`)

I read the adapters, the v0.17 Object schema + evolve helpers, the CLI, the web
surface, and the acoustic flow. **Two of OQ-33's premises are correct; one is
materially out of date.**

### 1.1 Confirmed premises

| OQ-33 claim | Code reality | Verdict |
|---|---|---|
| `MeshAdapter` sets `objects=[]` | `roomestim/adapters/mesh.py:154` — literal `objects=[]` with inline comment "v0.17: no auto-detection (OQ-33); use evolve_room_add_object()" | **CONFIRMED** |
| `ACEChallengeAdapter` sets `objects=[]` | `roomestim/adapters/ace_challenge.py:721` — same literal + comment | **CONFIRMED** |
| RoomPlan's auto-path is separate | `roomestim/adapters/roomplan.py:130-182` `_extract_objects()` does case-insensitive `category` substring matching on the ARKit sidecar (`"column"`/`"pillar"`/`"door"`/`"window"`); Mesh/ACE have **no** segmentation source to mimic this | **CONFIRMED** — RoomPlan auto-detection is a genuinely separate, source-specific path; it does not generalize to raw mesh |
| bbox-clustering algorithm "미안정" | No clustering code exists anywhere in `roomestim/`; candidate 2 is greenfield | **CONFIRMED** (unbuilt) |

`polycam.py` is a deprecated thin alias of `MeshAdapter` (`PolycamAdapter(MeshAdapter)`
+ DeprecationWarning), so "Polycam" inherits `objects=[]` — there is no separate
Polycam segmentation consumer today.

### 1.2 The out-of-date premise — candidate 3 is **already shipped**

OQ-33 lists resolution candidate **(3) "사용자 수동 annotation UI (ADR 0034 evolve
helper 활용)"** as a *future* option. In fact the manual-annotation path it
describes **already exists** as of v0.17 (ADR 0034 §A + D39):

- **Core evolve helpers (shipped):** `roomestim/edit.py:270` `evolve_room_add_object(room, obj)`
  and `:291` `evolve_room_remove_object(room, idx)`, both exported in `__all__`
  (`edit.py:461-467`). They wrap `evolve_room(room, objects=...)` over the frozen
  `RoomModel`. `roomestim/model.py:172-200` defines `Object`, `ObjectKind`,
  `DEFAULT_OBJECT_MATERIAL`.
- **Web manual-annotation UI (shipped):** `roomestim_web/object_add.py` is a full
  "Object Add Mode" — kind-specific input forms (column: anchor xyz + w/h/d +
  material; door/window: wall_index + wall-local anchor + w/h + material), input
  validation, and direct calls to `evolve_room_add_object` /
  `evolve_room_remove_object`. Tests: `tests/web/test_object_add_ui.py` (in the
  web lane).
- **Acoustic flow consumes objects (shipped):** `roomestim/reconstruct/predictor.py:208`
  `_objects_to_surfaces()` + `:454`/`:528` fold `room.objects` into the ISM/Sabine
  path (column → +5 surfaces, door/window → wall α override per D46), with the
  rationale string reporting `+N column surfaces` / `+N wall α overrides`.

**Consequence for the decision:** OQ-33's *actual* unresolved residual is **NOT**
"can a human add objects" (yes, since v0.17). It is the much narrower **"can an
adapter auto-extract objects from a non-RoomPlan source (raw mesh / ACE table)
without a human."** The two ways to do that — candidate 1 (proprietary Polycam
Pro segmentation API) and candidate 2 (greenfield, unstable bbox clustering) —
are exactly the high-risk options. The low-risk option (candidate 3) is done.

The only candidate-3 gap that remains is **CLI parity**: `roomestim` CLI has 5
subcommands (`ingest`/`place`/`export`/`run`/`edit`; `cli.py:31/54/103/158/219`)
and **no `add-object`** — manual annotation is web-only. That is a tiny
convenience, not the OQ.

### 2.2 D54 (recommended)

> **D54** — OQ-33 ("adapter object 자동 인식")의 **manual-annotation 부분은
> v0.17 에서 이미 충족됨**(core `evolve_room_add_object`/`evolve_room_remove_object`
> + web `object_add.py` Object Add Mode + predictor object-fold). 따라서 OQ-33 의
> 미해결 잔여를 **"non-RoomPlan adapter (Mesh/ACE) 의 무인 자동 객체 추출"** 로
> **범위 축소**하고, 그 잔여(후보 1 Polycam Pro API / 후보 2 bbox clustering)는
> **doc-only 로 v0.20 까지 재연기(re-defer)** 한다. 두 trigger (RoomPlan 외
> phone-scan 출처 자동 추출 요청 ≥ 1건; mesh-only object-GT fixture 도입)는
> **모두 미충족**이며, 두 자동-추출 후보 모두 0 user report 상태에서 가장 높은
> 리스크(미안정 클러스터링 / 비공개 API + non-CI)를 동반하므로 D26 의 YAGNI
> 규율에 정합한다. v0.20 은 **hard wall** 이다 (재-재연기 금지 — 그때까지도
> trigger 미충족이면 WONTFIX 로 정식 종결).
>
> **Rejected alternatives:**
> - **후보 1 (Polycam Pro API)** — 비공개 API 의존, Linux-CI 빌드 불가, fixture
>   없음. 0 user report 에 외부 의존을 들이는 것은 premature.
> - **후보 2 (bbox clustering)** — greenfield + self-described "미안정"; validation
>   ground-truth fixture 부재로 검증 불가한 heuristic 을 ship 하게 됨. D26 의
>   "보고 없는 추측 구현 금지" 위반.
> - **후보 3 을 "신규 작업"으로 취급** — 이미 ship 됨. 재구현은 중복.
> - **무기한 재연기 (wall 없음)** — D26 직접 위반.
>
> **Reverse-criterion (재연기 해제 조건):** 다음 중 하나라도 충족되면 v0.20 에서
> ADR 0037 (auto-recognition policy) 를 신규 작성하고 후보 1/2 중 검증 가능한
> 쪽을 구현한다 — (a) RoomPlan 외 출처(raw mesh/Polycam/ACE)에서 무인 객체 추출
> 요청 ≥ 1건, OR (b) object ground-truth 라벨이 붙은 mesh-only fixture 가
> repo 에 도입(후보 2 의 회귀 검증 기반 확보), OR (c) Polycam 이 안정적
> Linux-buildable segmentation export 를 공개(후보 1 의 CI 제약 해소).
> **Scope**: policy (no acoustic/schema code). **ADR ref**: ADR 0034
> §Status-update-v0.18.2 (residual 축소 + manual-path DONE 명시) + ADR 0030
> §Status-update-v0.18.2 Item X.
