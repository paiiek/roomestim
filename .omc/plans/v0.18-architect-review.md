## P0 / HIGH (must fix before execution)

### HIGH-1 — Aim-restoration idempotency hinges on an unstated branch-equivalence proof; add it as an explicit acceptance gate.
The whole D50 idempotency claim (Gate 17) rests on a subtle fact the plan never states:
the writer ALWAYS emits `x_aim_az_deg`/`x_aim_el_deg` even when `aim_direction is None`
(`layout_yaml.py:109-118` — None falls back to the "toward-origin" default direction
`(-pos.x,-pos.y,-pos.z)`). After Phase 2 the reader restores that into an EXPLICIT
non-None unit-vector aim. So the second write takes the `else` (explicit) branch
(`layout_yaml.py:113-116`) instead of the `if None` branch (`:109-112`).

Byte-equality therefore requires: `cartesian_to_pipeline(unit_of(-p)) ==
cartesian_to_pipeline(-p)` in emitted degrees. This holds because `cartesian_to_pipeline`
az/el are atan2 ratios (scale-invariant; magnitude discarded — `coords.py:38-48`). The
reasoning is CORRECT, but it is the linchpin of the release and is currently implicit.
**Fix**: in D50 / ADR 0036 §C, state the branch-switch (None→explicit) explicitly and
add a dedicated gate: "a layout with NO aim authored, written once (default-aim branch),
read (aim restored), written again (explicit-aim branch) → byte-equal." Gate 17 as
worded ("write→read→write") does not make the None-aim default-branch case explicit,
which is exactly the case `_cmd_export` hits on legacy layouts.

### HIGH-2 — `_cmd_export` byte-stability on PRE-v0.18 layouts is asserted but not gated.
Risk-row (ii) claims `_cmd_export` (`cli.py:323-357`) stays byte-stable after aim
restoration. This is the same None→explicit branch switch as HIGH-1, applied to the
existing export command on a layout file that has `x_aim_*` keys but was produced by
the default-aim path. The claim is correct (same proof), but there is NO acceptance gate
covering `roomestim export` round-trip byte-stability — Gate 5 only covers
`test_export_layout_yaml.py` (writer-only, never reads back). **Fix**: add a gate:
`read_placement_yaml(L)` → `write_layout_yaml(...)` → byte-equal to a re-write of the
same file, for a default-aim fixture. Without it, a regression in `_cmd_export` (a
shipped command) would pass all 24 gates undetected.

## MEDIUM

### MED-1 — `validate_placement` extraction must reproduce `write_layout_yaml`'s exact ORDER and RAISE semantics, not just the rule set.
The writer's contract (`layout_yaml.py:206` docstring + body) is order-sensitive:
Step 1 R10 → Step 2 `placement_to_dict` (which itself raises for WFS-without-alias,
`:159-163`) → Step 3 finite-sweep → Step 4 schema. The plan's `validate_placement`
sketch (plan:439-458) reorders: it does R10 in one try/except, then bundles dict-build +
finite + schema in a single `except Exception`. Two divergences: (a) the WFS-missing-alias
ValueError from `placement_to_dict` currently aborts BEFORE write with a specific message;
folding it into a generic error-string list changes its surfacing. (b) `write_layout_yaml`
RAISES (ValidationError/ValueError propagate); `validate_placement` RETURNS strings. The
plan says the writer should "call this helper and raise the first error" (risk row 2) —
but the writer's existing exceptions are *typed* (`ValidationError`, `ValueError`), and a
"raise RuntimeError(first_string)" wrapper would change exception type, breaking any caller
that catches `jsonschema.ValidationError`. **Fix**: specify that `write_layout_yaml`
retains its own raise path UNCHANGED and `validate_placement` is a SEPARATE non-raising
collector (duplicating the rule invocations, not the control flow), OR that the refactor
preserves exact exception types. Gate 21 (byte-equal output) does NOT catch exception-type
drift. Recommend the "separate collector, writer untouched" option — it is the safer of the
two the plan itself lists as rollback, and avoids touching a shipped raise contract.

### MED-2 — Round-trip "Level 1" silently drops `id` and the VBAP/WFS algorithm inference is lossy.
The reader infers `target_algorithm` from `regularity_hint`/`x_wfs_f_alias_hz`
(`placement_yaml_reader.py:45-54`): `LINEAR`→WFS, else→VBAP. So a DBAP or AMBISONICS
layout round-trips as "VBAP", and the per-speaker `id` field (writer emits `id` ==
`channel`, `:127`) is dropped on read. The D50 contract lists `target_algorithm ==` as
preserved — but it is only preserved for the VBAP/WFS subset. **Fix**: D50 should
explicitly scope the algorithm-preservation claim to {VBAP, WFS-via-LINEAR} and note DBAP/
AMBISONICS collapse to VBAP as a known Level-1 limitation (or add an OQ). This is pre-existing
reader behavior, not introduced by v0.18, but the plan formalizes a "contract" that overstates it.

### MED-3 — CLI `--del-deg` asymmetry is a real ergonomic wart the plan accepts without a cleaner alternative considered.
The plan lands `--daz / --del-deg / --ddist` (the `del` reserved-word dodge, plan:512-515).
Mixing bare `--daz` with suffixed `--del-deg` is incoherent. A cleaner surface the plan
did not weigh: name all spherical deltas with consistent unit suffixes (`--daz-deg /
--del-deg / --ddist-m`) so the oddball is not singular, OR use `--delev` (elevation, no
reserved-word clash, no suffix). **Fix**: pick one consistent scheme. Low effort, improves
the permanent CLI surface.

  one-line "verified non-path" note for completeness.

## Antithesis (steelman against the favored direction)
**The cheaper alternative the plan dismisses too quickly: do NOT restore aim in the
shared reader — restore it only in the edit path.** D50 makes `read_placement_yaml`
start returning non-None `aim_direction` for EVERY caller, including the shipped
`roomestim export` command, purely so the (new, narrower) nudge path preserves aim.
That is widening a shared function's output contract to serve one new consumer — the
exact "load-bearing risk" the planner flagged. A lower-blast-radius design: keep
`read_placement_yaml` byte-for-byte behavior-stable (aim stays None) and add an opt-in
`read_placement_yaml(p, restore_aim=True)` (which the plan already lists as its own
rollback, risk row 1) — making restore the *edit-path-only* behavior from day one
rather than the rollback. This eliminates HIGH-2 entirely (export untouched) and shrinks
HIGH-1 to the edit path. The plan rejects opt-in because "D50 weakened," but D50's goal
(nudge preserves aim) is fully met by opt-in; only the *aesthetic* of "reader is now
lossless" is lost. **Tradeoff tension**: default-restore gives a cleaner one-true-reader
story and makes future consumers automatically aim-aware, at the cost of a wider blast
radius on a shipped command; opt-in restore is surgically safe but leaves two read
behaviors. Given roomestim's verify-each-step discipline and that aim is currently
*silently dropped everywhere*, default-restore is defensible — but the plan should
explicitly record WHY it chose the wider-blast-radius option over its own rollback, not
treat opt-in as merely a fallback. **Recommended synthesis**: ship default-restore (D50
as written) BUT add HIGH-2's export round-trip byte gate as a hard precondition, so the
wider contract is locked by a regression test rather than by prose reasoning.
