## Findings

### C-1 [P1 / MAJOR] — D50 "Level 1 contract" overstates fidelity: `target_algorithm` does NOT round-trip for DBAP
- **Claim**: §2 D50 + §3 Phase5 `test_channel_regularity_wfs_preserved` formalize `read(write(r)) == r` over `(position, channel, regularity, target_algorithm, wfs_f_alias_hz, aim)`. Line 179-181 explicitly lists `target_algorithm ==`.
- **Evidence (REFUTED)**: reader infers `target_algorithm` only as WFS (x_wfs_f_alias_hz present OR regularity==LINEAR) else **"VBAP"** (placement_yaml_reader.py:46-54). A DBAP result (regularity_hint="IRREGULAR", dbap.py:326) round-trips as `target_algorithm="VBAP"`. Empirically proven: `original=DBAP → round-tripped=VBAP, MATCH? False`. The plan's only round-trip-of-algorithm test is WFS, which dodges this.
- **Why it matters**: The plan elevates a lossy reader into a "formal contract" and asserts a guarantee that is false. An executor writing `test_channel_regularity_wfs_preserved` honestly (or a future DBAP case) will either pass vacuously or hit a failure the contract said couldn't happen. This is exactly the "honesty leak" the project's D27/D35 discipline forbids.
- **Fix**: Either (a) scope D50 explicitly: "`target_algorithm` round-trips for WFS and VBAP; DBAP→IRREGULAR collapses to VBAP on read (pre-existing reader limitation, NOT fixed in v0.18) — documented limitation, add to §Round-trip 충실도 표 exclusions + new OQ or note", OR (b) fix the reader to store/restore `target_algorithm` (e.g. an `x_target_algorithm` extension key) within v0.18 scope. (a) is cheaper and consistent with the Level-1/Level-2 honesty framing already in the plan. Do NOT leave the contract claiming unconditional `target_algorithm ==`.

### C-2 [P1 / MAJOR] — `--del` CLI flag has a three-way internal contradiction (executor must guess)
- **Claim**: Phase 4 / §1 Scope table / risk table specify the spherical-elevation-delta flag inconsistently.
- **Evidence**: line 83 + line 472 show `--del` (broken: `args.del` is a Python syntax error); line 489 `_cmd_edit` reads `del_deg=args.delv`; lines 513-515 + 665 decide on **`--del-deg`** → `args.del_deg`; line 533 web handler uses param name `delv`. Four spellings (`--del`, `delv`, `del_deg`, `--del-deg`) for one concept.
- **Why it matters**: The executor cannot know whether the argparse `dest` is `delv` or `del_deg`, nor which long-option string to register. `_cmd_edit` line 489 (`args.delv`) directly contradicts the resolution at 513 (`args.del_deg`). This is the single most likely point where the executor stalls or ships a broken CLI.
- **Fix**: Pin one spelling everywhere. Recommend `--del-deg` with `dest="del_deg"`, and update line 489 to `del_deg=args.del_deg`, line 472/83 to `--del-deg`, and the web `_on_nudge_speaker` signature param (line 533) to a non-`del` name (e.g. `del_deg`). Also confirm the matching `--daz`/`--ddist` map to `nudge_speaker(daz_deg=, ddist_m=)` — the kwarg names differ from the flag names, so the executor needs the explicit mapping (`--daz`→`daz_deg`, `--ddist`→`ddist_m`).

## What's Missing (gaps)
- **Reader robustness**: `_aim_from_speaker` requires BOTH `x_aim_az_deg` AND `x_aim_el_deg`. Plan handles "both absent → None" but not "exactly one present" (malformed/hand-edited layout). Behavior = None (silently ignores a present key). Acceptable but should be a documented choice + a round-trip test asserting partial keys → None (not a crash).
- **`nudge_speaker` el clamp**: risk table notes el can exceed ±90 (non-physical) but defers to "web min/max=−90/90" — CLI has no guard. `evolve_placement` finite-check passes el=200° (still finite). Plan acknowledges this and defers clamp to v0.18.1; acceptable as a documented known-gap, but the CLI help should warn "no el clamp" or `nudge_speaker` should reject |result el| > 90. Currently underspecified whether CLI silently accepts non-physical el.
- **DBAP nudge end-to-end**: with C-1 unfixed, `roomestim edit` on a DBAP layout reads it as VBAP, nudges, re-writes as VBAP. The regularity_hint (IRREGULAR) is preserved so engine validation still passes, but the in-memory `target_algorithm` is wrong. No test covers `roomestim edit` on a DBAP fixture.
