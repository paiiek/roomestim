## §8 Schema marker decision

`__schema_version__` UNCHANGED at `"0.1-draft"` at v0.11.

**Rationale**:

1. ADR 0016 §Reverse-criterion binds the schema flip to A10b in-situ
   capture (in-situ ALWAYS overrides substitute).
2. ADR 0018 §Reverse-criterion further tightens this to require ≥ 2
   SoundCam rooms (lab + conference) returning to A11 PASS-gate under
   paper-faithful material maps AND ≥ 3 captures per D2 (and ADR 0018
   §References tightening).
3. v0.11 might satisfy condition (lab + conference at PASS) IF
   executor §2.4 picks sub-branch A. But D2 still requires ≥ 3
   captures, and v0.11 still has only 2 substitute rooms (lab + conference);
   living_room is REMOVED. The third room would need to be a new
   capture OR a third SoundCam room with paper-authoritative dims
   (none exist at v0.11 — paper §A.2 does not publish authoritative
   dims for living_room or any other room beyond lab + conference).
4. Therefore: even if lab returns to PASS, the marker MUST NOT
   re-flip at v0.11. The re-flip is bound to a future v0.12+ event:
   either (a) A10b in-situ capture lands (lab_real.usdz fixture available);
   or (b) a third SoundCam-equivalent substitute room with paper-authoritative
   dims emerges.
5. **Concrete invariant**: `__schema_version__ == "0.1-draft"` byte-equal
   under v0.11.

**Reverse-criterion**: if at v0.11 ship time BOTH (a) executor §2.4 picks
sub-branch A (lab PASS) AND (b) user lands an A10b in-situ capture
within the v0.11 cycle (lab_real.usdz fixture available), schema marker
MAY re-flip at a successor v0.11.1 patch or v0.12.0 minor — NOT at
v0.11.0 itself. The reverse-criterion is documented in ADR 0019
§References (cross-referencing ADR 0016 §Reverse-criterion).

### §9.3 D25 NEW — Doc-ahead-of-implementation pattern (v0.11 in-situ protocol DOC precedent)

- **Date**: 2026-05-11
- **Decision**: v0.11 codifies the doc-ahead-of-implementation pattern: when a future capability (A10b in-situ capture) is user-volunteer-only and has no committed timeline, but the implementation protocol benefits from being recorded BEFORE the capability lands, the protocol DOC is shipped at a planning-stable version (v0.11) under a `docs/protocol_*.md` path WITHOUT an accompanying ADR (the ADR is reserved for the architectural decision that already exists — ADR 0016 in this case). When the capability lands (v0.12+), a successor ADR (e.g., ADR 0021) codifies the capture-process invariants.
- **Drivers**: OQ-12a status-update at v0.10.1 stated "v0.11 will ship in-situ protocol DOC only (no capture commitment)". v0.11 implements that; D25 codifies the pattern for future "doc-ahead" cases (e.g., live-mesh extraction protocol could ship at v0.12 ahead of OQ-13e closure).
- **Why this pattern, not alternatives**: (a) "Wait until A10b capture lands, then write the protocol DOC retroactively" — rejected because protocol-after-implementation is the v0.9 honesty-leak failure mode (placeholder-as-measured). Writing the protocol first sets expectations BEFORE empirical data; the protocol is therefore not back-fitted to make the implementation "pass". (b) "Ship the protocol DOC as an ADR" — rejected because the operational protocol is not an architectural decision; ADR 0016 already records the in-situ-overrides-substitute architecture.
- **Reverse-criterion**: if v0.12+ user-volunteer captures fail because the protocol is too thin (missing edge cases), expand the DOC inline at v0.12. If the protocol DOC starts contradicting an established invariant (e.g., specifying ≥ 4 corners while a target room is L-shaped with 6 corners), patch under a successor D26.
- **Cross-references**: D22 (audit-trail discipline; doc-and-implementation parallel), ADR 0016 (architectural decision), OQ-12a (status-update at v0.11; resolution-candidate unchanged on capture commitment), `docs/protocol_a10b_insitu_capture.md`.
