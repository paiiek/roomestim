## 1. Principles (decision-guiding constants)

1. **Additive over disruptive** — every change keeps existing call-sites green. Default parameter values reproduce v0.1 byte-for-byte (A12 idempotency stays intact).
2. **Honour D8 boundary** — Stage-2 schema flip and physical lab capture remain post-autopilot human-gated. v0.1.1 does NOT promote Stage-2.
3. **Read-only on spatial_engine** — sibling repo source is untouchable at v0.1. The harness may *invoke* built binaries and *read* schema/configs only.
4. **Evidence over claims** — every change lands with a test that fails today and passes after. No "should work" without an assertion. Where a test threshold cannot be derived a-priori, the test is reframed as **characterisation** (asserts only invariants, prints data for diagnostic value) rather than fabricated.
5. **Default-CI lane stays green** — `pytest -m "not lab"` must pass on every commit; `@pytest.mark.lab` SKIP is acceptable but must be graceful.

## 2. Decision Drivers (top 3, ranked)

1. **Faithfulness to D8** — autopilot exit boundary cannot be violated (A10 stays human-gated; Stage-2 stays post-autopilot).
2. **Minimum surface area** — v0.1 shipped 18/18; the closeout should not introduce regression risk in unrelated modules. Additive-only edits to `roomestim/place/vbap.py` + new test files are preferred over refactors.
3. **Honest accounting** — claims about "closure" must match actual artifact type. A deferral is not a closure; an audit is not a closure; a characterisation test is not a numerical bound test. The plan and the RELEASE_NOTES must say the same thing.

## 3. Viable Options

### Option A — Honest closeout (RECOMMENDED, REVISED)

**Scope**: Closes 2/4 addressable items (phase-offset gap → tested; DBAP geometry-dependence → characterised), defers 1/4 with explicit decision (A2/A15 → spatial_engine v0.2), audits 1/4 (A10 fixture path).

- **Pros**: Honest framing avoids the iter-1 "4/4 closed" overclaim flagged by Critic M2; perf_verification §Limitations shrinks from 6 entries to 2 substantive open items (RT60 model accuracy; A10 physical capture) + 2 documented-state items (A2/A15 deferred to engine v0.2; DBAP characterised).
- **Cons**: ~5 file edits, 4 new test files (one is a frozen golden fixture, not a test module; net 3 new test modules), ~5 hours of focused work.

### Option B — Minimum closeout (just C++ parity wire-up)

**Scope**: Set SPATIAL_ENGINE_BUILD_DIR via fixture; ship as v0.1.0.1.

- **Pros**: Smallest diff, fastest ship.
- **Cons**: **Rejected** — the named binaries (`layout_loader_smoke`, `coords_parity_harness`) do **not exist** in `/home/seung/mmhoa/spatial_engine/core/build/` (verified 2026-05-04). Wiring an env var alone changes nothing; tests still skip. Without a parity decision, this option ships zero behavioural change. Phase-offset and DBAP-noise gaps remain open.

### Option C — Phase-offset only

**Scope**: Just add `phase_offset_deg` and the lab-fixture position-subset match test.

- **Pros**: Solves the most user-visible gap (lab regeneration); 1-file change + 1 test.
- **Cons**: Leaves DBAP geometry-dependence uncharacterised. DBAP IS geometry-dependent (consumes `mount_surfaces` derived from floor_polygon) AND structurally non-smooth (`_greedy_max_min_select` uses argmax tie-break — `dbap.py:215–247`). Skipping the characterisation test means the plan ships no evidence that DBAP positions don't NaN/inf or leave the mount surfaces under perturbation.

### Decision: Option A (revised)

Option A revised. Driver-3 (honest accounting) replaced iter-1's "falsifiable closure" because the iter-1 phrasing pushed Step 4 toward fabricated thresholds (Critic M1). The new framing distinguishes *closure* (test asserts a derivable bound) from *characterisation* (test asserts only invariants, plus diagnostic output) from *deferral* (decision document). All three are legitimate v0.1.1 outcomes; conflating them is not. Option B rejected: binaries absent. Option C rejected: leaves DBAP non-smoothness silent — exactly the risk surface S3 was supposed to cover.
