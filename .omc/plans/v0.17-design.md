## 1. 목표 — 사용자 4-제안 ①(나머지)/③(나머지) land

v0.16.1까지 RoomModel은 `surfaces: list[Surface]` (벽/바닥/천장) 만 표현 가능
했으며, phone-scan 출력에는 자주 등장하는 **기둥 / 도어 / 창문** 3 객체가
누락된 채로 흡수되어 음향 예측이 체계적으로 편향되었다 (예: glass 창문 → wall
으로 라벨링 → α=0.05 vs 실제 α≈0.04, 큰 차이 X 이지만 ACEChallenge conference
fixture는 ratio 5.05 가 부분적으로 이 누락에서 기인). 동시에 export 출력은
`layout.yaml` (engine schema) + `room.yaml` (단순 직렬화) 2 포맷 뿐 — 사용자가
3D 도구 (Blender / Apple Vision Pro Preview / Reality Composer / three.js
viewer) 로 import 하려면 별도 변환이 필요했다.

v0.17에서 (a) `Object` frozen dataclass + `RoomModel.objects` 필드, (b) RoomPlan
adapter 의 column/door/window 카테고리 매핑, (c) Web UI "Object Add Mode" 3종
(2D 블루프린트 클릭 → column/door/window 추가), (d) `predictor.py` ISM/Eyring
경로의 객체 흡수 (column = 4 추가 표면; door = wall α override 영역; window =
glass α override 영역), (e) USDZ export (`roomestim/export/usd.py` via `usd-core`),
(f) gLTF export (`roomestim/export/gltf.py` via 기존 `trimesh` dep) 6 surface
를 동시 land한다. `__schema_version__` 은 `"0.2-draft"` 로 bump하되 구 schema
(`"0.1-draft"`) YAML 은 `RoomModel.objects = []` 로 backward parse (D44 정책).
RoomModel frozen invariant 보존 (모든 mutation은 `dataclasses.replace` chain).
ADR 0009 ISM ≥ Eyring invariant 보존 (column 추가 surface 시 재검증; D47 회귀
lock 50 seeds × 3 object kind).

### 비-목표 (영구 거부)

  "door" | "window"`). 사용자가 "speaker stand / chair" 등 요청 시 별도 enum
  멤버 추가 결정은 ADR 0034 §Reverse-criterion 발동 (≥ 3 사용자 요청 또는
  acoustic 영향 ≥ 0.1s rt60).
- Mutable `RoomModel.objects` (`field(default_factory=list)` 는 의도된 immutable
  default — `dataclasses.replace` 시 새 list instance; D39 `evolve_room` 시그니처

- USDZ via `pxr` (Pixar OpenUSD C++ SDK) — Python wheel 부재, Linux 빌드 5min+

## 2. 결정 (신규 D 4건 + 신규 ADR 3건; OQ 3건 신규 deferral)

### D-decisions

- **D44 NEW — Schema bump `"0.1-draft"` → `"0.2-draft"` + backward parse 보장**
  - 선택 근거: `RoomModel` 에 `objects: list[Object] = field(default_factory=list)`
    필드 추가는 구 schema YAML (필드 부재) 의 deserialization 을 깨지 않아야 함.
    YAML reader 는 `schema_version` 분기를 명시적으로 두어 `"0.1-draft"` 입력 →
    `objects=[]` 자동, `"0.2-draft"` 입력 → `objects` 키 의무 (빈 list 라도
    명시).
  - vs (A) MAJOR bump `"0.1-draft"` → `"1.0"` (stable schema 명시): pre-1.0
    소프트웨어가 stable 약속을 너무 일찍 함 → 향후 mesh segmentation /
    speaker_array 추가 시 또 다른 MAJOR 강제 → reject. vs (B) `"0.2-objects"`
    semantic 이름: schema_version 은 incremental 식별자이지 feature 라벨이 아님
    (`"0.3-acoustic-furniture"` 등 chain 불편) → reject. vs (C) `"0.2-draft"`
    숫자 incremental + 본문 변경 명세는 schema 파일 자체에 위임 → **채택**.
  - 구 schema 호환 정책 (room_yaml_reader.py):
    ```python
    schema_version = data.get("schema_version", "0.1-draft")
    if schema_version == "0.1-draft":
        objects = []  # backward: 0.1-draft 에는 objects 필드 부재
    elif schema_version == "0.2-draft":
        objects = [_parse_object(o) for o in data.get("objects", [])]
    else:
        raise ValueError(f"Unsupported schema_version: {schema_version}")
    ```
  - 신 schema export 정책 (room_yaml.py): 항상 `schema_version: "0.2-draft"`
    write + `objects:` 키 무조건 emit (빈 list 라도) — round-trip determinism.
  - 역방향 기준: 구 schema 외부 consumer (`spatial_engine` 등) 가 0.2-draft
    YAML 에서 unknown field 로 fail 하는 경우, `roomestim export --schema 0.1`
    flag 도입 검토 (v0.17.1 patch 후속 또는 OQ-36 NEW). 현재는 export 가
    `layout.yaml` (engine schema) 와 `room.yaml` (직렬화) 분리 — `room.yaml`
    소비자는 roomestim 자체 + 사용자 도구 (드물게 외부) 이므로 위험 낮음.

- **D45 NEW — USDZ backend: `usd-core` (Pixar 후원 PyPI wheel) vs `pyusd`
  (deprecated) — `usd-core` 채택**
  - 선택 근거: `pyusd` 는 PyPI 마지막 release 2019, GitHub archived. `usd-core`
    가 현행 Pixar OpenUSD 의 공식 PyPI 배포 (`pip install usd-core` →
    `from pxr import Usd, UsdGeom, Sdf` 동일 API), Linux/macOS wheel 제공,
    Python 3.10+ 호환 검증. Vision Pro / Reality Composer / Apple Preview.app
    모두 USDZ 표준 호환.
  - vs (A) `pyusd`: deprecated → reject. vs (B) `usd-core`: 공식 후원 + 활성
    유지 → **채택**. vs (C) `pxr` C++ SDK 직접 빌드: Linux wheel 부재 + 5min
    빌드 부담 + Docker 이미지 비대 → reject.
  - pyproject.toml 변경: `[project.optional-dependencies] usd = ["pyusd"]` →
    `usd = ["usd-core>=24.0"]`. `pyusd` 는 v0.16.1까지 정의만 있었고 실제
    사용처가 없었으므로 (export/usd.py 부재) breaking 영향 0.
  - Import-time graceful degradation: `try: from pxr import Usd, UsdGeom, Sdf;
    except ImportError as e: raise ImportError("USDZ export requires
    `pip install roomestim[usd]`") from e` — `roomestim/export/usd.py` 모듈
    top-level (lazy import 패턴; CLI 가 다른 포맷 사용 시 import 안 됨).
  - 역방향 기준: `usd-core` 가 archive 되거나 Apple 이 별도 spec 으로 분기 시
    `pxr` 직접 빌드 또는 USDZ writer 자체 구현 (USDZ = uncompressed zip + USD
    binary; spec 공개 — 최소 mesh writer ~300 LoC) 검토.

- **D46 NEW — Object 음향 통합 정책: column = 표면 추가, door/window = 벽 α
  override 영역 (혼합 절충)**
  - 선택 근거: column 은 룸 안쪽 standalone 객체 (4 측면 + top — 음향적으로
    완전한 추가 표면; absorption coefficient 자체 보유; ISM α 계산에 직접
    참여). door/window 는 벽-내 부분 영역 (벽 표면적의 일부를 다른 α 로 대체;
    별도 standalone surface 가 아님 — 벽이 차지하는 공간을 공유).
  - column 처리: `predictor._objects_to_surfaces(objects)` 가 column 객체를
    `[Surface(kind="wall", polygon=column_side_polygon, material=col.material), ...]`
    리스트 (4 면) + `Surface(kind="ceiling", polygon=column_top_polygon,
    material=col.material)` 1면 = 5 추가 surface 로 변환 후 기존
    `_shoebox_surface_areas_and_alphas` 에 머지. ISM α 가 area-weighted average
    이므로 column 면적이 작으면 영향 미미 (lab_room 중앙 column 0.3×0.3×2.85
    면적 3.42 m² / lab 전체 표면적 ~80 m² = 4.3% 가중치 → rt60 영향 < 0.05s
    예측).
  - door/window 처리: `predictor._objects_to_wall_alpha_overrides(objects)` 가
    door/window 객체를 `{wall_index: [(area_m2, override_material), ...]}` dict
    로 변환 → `_shoebox_surface_areas_and_alphas` 가 wall surface 별 효과적 α
    를 (벽 α × (1 - override_area_fraction) + override α × override_area_fraction)
    공식으로 재계산. door material = WALL_PAINTED 면 영향 0; window material =
    GLASS 면 영향 (glass α=0.04 vs wall_painted α=0.05 차이 작음 — 500Hz 기준
    rt60 영향 < 0.02s lab_room).
  - vs (A) column 도 alpha override 처리 (벽 한 면을 jutting 한 column 영역
    으로 분할): 기둥은 벽에서 떨어져 있을 수 있음 (룸 중앙 standalone
    column) → 잘못된 모델 → reject. vs (B) door/window 도 standalone surface
    추가 (벽 surface + door surface 둘 다 ISM 에 참여): 같은 공간에 두 표면
    이 겹쳐 ISM image source 가 중복 계산 → reject. vs (C) 혼합 절충 (column
    = 표면 추가; door/window = wall α override) → **채택**.
  - ADR 0009 invariant 영향 검증: column 5 추가 surface 가 image-source order
    50 에서 ISM 결과를 변화시키지만, eyring 도 동일 surface 리스트로 재계산
    하므로 invariant `ism_rt60 ≥ eyring_rt60 - 1e-6` 는 수학적으로 보존됨
    (둘 다 같은 α 분포 사용). D47 회귀 lock (50 random seeds × 3 object kind
    × lab_room) 으로 확인.

- **D47 NEW — Object 추가 시 ADR 0009 invariant + ADR 0030 cascade 회귀 lock
  (50 random seeds × 3 object kind)**
  - 선택 근거: D43 (v0.16.0) 가 material 변경에 대한 invariant 회귀 lock 을
    50 seeds × 10 materials × 6 surface 로 land. v0.17 에서 object 추가는
    surface 수를 변경 (column +5, door/window α 영역 변경) → invariant 재검증
    필요. material shuffle 만으로는 surface 수 변동 케이스 미커버.
  - 결과: `tests/test_objects_acoustic_invariant.py` NEW — 50 random seeds ×
    {column 1개 base+height random / door 1개 wall+area random / window 1개
    wall+area random} = 150 instance × `predict_rt60_default` 호출 × invariant
    `ism_rt60 ≥ eyring_rt60 - 1e-6` 검증 + `default_predictor_name in {"image_source",
    "eyring"}` cascade 검증.
  - 부수 효과: column 추가 후에도 `is_rectilinear_shoebox(room)` 는 floor_polygon
    만 판정하므로 True 유지 (column 은 polygon 변경 X) → ISM 분기 유지 → 정책
    안정성. door/window 추가는 surface 수 자체 변경 X (벽 α override 만) → ISM
    분기 영향 0.

### ADR refs

- **ADR 0034 NEW — Object schema (Column/Door/Window) 정책** (~140줄)
  - §Context: phone-scan 출력 RoomModel 은 v0.16.1까지 6 surface (4 wall +
    floor + ceiling) shoebox 만 정확히 표현. 실제 룸의 기둥/도어/창문은 모두
    벽으로 흡수되어 음향 예측에 체계적 편향. ACEChallenge conference fixture
    가 ratio 5.05 인 부분 원인.
  - §A Scope: `Object` frozen dataclass — `kind: Literal["column", "door",
    "window"]`, `anchor: Point3` (column = base center; door/window = bottom-left
    corner), `width_m: float`, `height_m: float`, `depth_m: float` (column only;
    door/window 는 0), `wall_index: int | None` (column = None standalone;
    door/window = 부착 벽 index), `material: MaterialLabel` (default per kind:
    column = `WALL_CONCRETE`, door = `WALL_PAINTED`, window = `GLASS`).
  - §B RoomModel 통합: `objects: list[Object] = field(default_factory=list)`
    추가; backward parse 보장 D44.
  - §C 음향 통합: D46 — column = 표면 추가 (5 surface: 4 측면 + 1 top); door/
    window = wall α override 영역 (area-weighted blend).
  - §D Invariant: D47 — ADR 0009 `ism_rt60 ≥ eyring_rt60 - 1e-6` 유지 (50 seeds
    × 3 kind 회귀 lock); ADR 0030 cascade `default_predictor_name` 유지.
  - §E Reverse-criterion: 사용자가 `kind` 무제한 확장 요청 (≥ 3건) 또는 음향
    영향 ≥ 0.1s rt60 인 새 kind 발견 시 enum 확장 + ADR 0034 §Status-update
    추가. 곡선/원형 객체 요청 시 polygon 근사 정책 (n=16) 도입.
  - §References: ADR 0002 (RoomModel shape), ADR 0008 (octave-band α), ADR 0009
    (ISM ≥ Eyring), ADR 0019 (enum extension cadence precedent — MELAMINE_FOAM),
    ADR 0031 (material override — door/window default material 결정과 일관),
    D44 / D46 / D47.

- **ADR 0035 NEW — Mesh export 정책 (USDZ via usd-core; gLTF via trimesh)** (~120줄)
  - §Context: v0.16.1까지 export 는 `layout.yaml` (engine schema) + `room.yaml`
    (직렬화) 만. 3D 도구 import 를 위한 표준 mesh format 부재 — Vision Pro /
    Blender / three.js 사용자가 매번 변환 스크립트 작성.
  - §A Scope: USDZ (Apple Vision Pro / Preview.app / Reality Composer 표준) +
    gLTF (Khronos 표준; Blender / three.js / Unity / Unreal 광범위 지원); 두
    포맷 모두 polygon-mesh + per-surface material color (PBR baseColor) 만 포함;
    acoustic α 메타데이터는 sidecar `.json` (선택).
  - §B Backend: D45 — USDZ via `usd-core` (`pyusd` deprecated); gLTF via 기존
    `trimesh>=4.0` (core dep — 추가 의존성 0).
  - §C Geometry mapping:
    - RoomModel surfaces (wall/floor/ceiling) → Mesh face (triangulated CCW per
      ADR 0002 polygon convention; shapely triangulation 또는 fan triangulation).
    - RoomModel objects (column) → Mesh primitive (5 face per column; box prim
      또는 explicit triangulated polygon).
    - RoomModel objects (door/window) → 벽 mesh 의 sub-region 으로 별도 face
      (USDZ subset prim 또는 gLTF sub-mesh) + material override.
    - listener_area → 별도 sub-prim (semi-transparent green; α=0.3).
    - speakers (placement.speakers) → 작은 sphere primitive (radius 0.05m;
      channel label 은 USD `displayName` / gLTF `node.name`).

    z=forward) 와 동일 — 좌표 변환 불필요. USDZ stage `upAxis="Y"` 명시 +

    1차 시도는 hash-equal 약화 + round-trip equality (write → read → mesh

  - §G Reverse-criterion: 사용자가 다른 mesh format (`.fbx`, `.obj`, `.dae`)

    ```python
    def _objects_to_surfaces(objects: list[Object]) -> list[Surface]:
        extra: list[Surface] = []
        for obj in objects:
            if obj.kind != "column":
                continue
            cx, cy, cz = obj.anchor.x, obj.anchor.y, obj.anchor.z
            hw, hd = obj.width_m / 2, obj.depth_m / 2
            base_corners = [
                Point3(cx - hw, cy, cz - hd),
                Point3(cx + hw, cy, cz - hd),
                Point3(cx + hw, cy, cz + hd),
                Point3(cx - hw, cy, cz + hd),
            ]
            top_corners = [Point3(p.x, p.y + obj.height_m, p.z) for p in base_corners]
            # 4 side faces (wall kind) — CCW from outside of column
            for i in range(4):
                j = (i + 1) % 4
                extra.append(Surface(
                    kind="wall",
                    polygon=[base_corners[i], base_corners[j], top_corners[j], top_corners[i]],
                    material=obj.material,
                    absorption_500hz=MaterialAbsorption[obj.material],
                    absorption_bands=MaterialAbsorptionBands[obj.material],
                ))
            # 1 top face (ceiling kind)
            extra.append(Surface(
                kind="ceiling",
                polygon=top_corners,
    ```
