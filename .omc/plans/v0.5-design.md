# roomestim v0.5 Release — Design Plan

- **Date**: 2026-05-06 (drafted) / 2026-05-07 (scope locked)
- **Author**: oh-my-claudecode:planner
- **Predecessor**: v0.4.0 (`5b41beb` — Eyring predictor + ACE audit findings)
- **Status**: SCOPE LOCKED — partial-A (dims-only) + B (MISC_SOFT enum). See §0a.
- **Mode**: SHORT (additive enum + adapter-data dimensional patch; no architecture
  redesign, no Stage-2 schema flip, no coefficient revision).

This plan covers the four DEFERRED audit items recorded in v0.4
(`.omc/plans/v0.4-audit-findings.md`):

- **Finding 1** — `ACE_ROOM_GEOMETRY` byte-audit vs Eaton 2016 TASLP Table I.
- **Finding 3** — Lecture_2 ceiling material reassignment hypothesis.
- **Finding 4a** — `MaterialAbsorptionBands` coefficient revision for `wall_painted`,
  `wood_floor`, `ceiling_drywall`.
- **Finding 4b** — `MISC_SOFT` MaterialLabel enum extension (schema-impacting).

It also keeps a contingency lane for cases where Finding 1's external dependency
(canonical paper) does not resolve in time. Three scope scenarios are written end-to-end
so the human reviewer can pick once the research report lands.

The local tag `v0.5.0` is local-only (matches v0.2.0 / v0.3.0 / v0.3.1 / v0.4.0
precedent: only v0.1.1 lives on the remote, per D11).

## §0a Research outcome → locked scope (2026-05-07)

The parallel `cwm:websearchwithme` report at `.omc/research/ace-table-i-acquisition.md`
landed with a partial result:

- **Dimensions** (L×W×H) for all 7 rooms: **AVAILABLE** in arXiv:1606.03365 Table 1

  vs `ACE_ROOM_GEOMETRY`: **Office_2** (arXiv 3.22 × 5.10 × 2.94 m vs roomestim

**Resolved open questions (§3 / §8 mirror)**:

- **OQ-2**: representative-not-verbatim with honesty marker — **default APPROVED**.

- **OQ-5**: ADR ordering inverted from initial draft to put the F1-partial headline

- **L/W convention**: keep roomestim's "longer-dimension-as-L" — patch Office_2 only

### Scenario A — "Wait for Table I" (paper-available branch)

**Trigger**: `.omc/research/ace-table-i-acquisition.md` reports a viable acquisition
path (e.g., legitimate IEEE Xplore access, institutional library, author-provided
preprint, ACM Open Access mirror). The reviewer extracts Table I to a verbatim local
artefact at `/tmp/ace_corpus/eaton_2016_table_i.csv` (or under
`tests/fixtures/ace_eaton_2016_table_i.csv` if redistribution licence allows).

**Goal**: Land Finding 1 byte-audit and the F3/F4a follow-ups whose direction Finding 1
disambiguates.

**Step-by-step**:

1. **Capture Eaton 2016 Table I locally** (single committed text fixture).
   - File: `tests/fixtures/ace_eaton_2016_table_i.csv` (or `tools/ace_table_i.txt` if
     licence prohibits redistribution — in that case keep an out-of-tree path

     measured_t60_500hz_s` (mirror the columns Eaton actually publishes; NOT a
     paraphrase of `ACE_ROOM_GEOMETRY`).

     transcription date.
2. **Add gated audit test** `tests/test_ace_geometry_audit.py`
   (mark `e2e` + `network`-style env-gate `ROOMESTIM_E2E_DATASET_DIR` so default lane
   is unaffected).
   - For each row in the local Table I copy: assert
     `ACE_ROOM_GEOMETRY[row.room_id]` matches `(L, W, H)` within `±0.01 m`,
     and that the assigned material strings match the paper's labels (with a
     declared crosswalk for any synonym, e.g. "carpet" ↔ "carpet_thin").
   - Audit emits a markdown delta report at
     `docs/ace_geometry_audit_2026-05-XX.md` (auto-regenerated; same pattern as the
     existing perf doc).
3. **Patch `ACE_ROOM_GEOMETRY` for any disagreement found**.
   - Each delta is one diff hunk in `roomestim/adapters/ace_challenge.py`.
   - Crucially: this is **adapter-only**. `MaterialAbsorptionBands` and
     `MaterialAbsorption` are NOT touched in this step. d-inv-3 byte-equality of all
     existing default-lane tests still holds (no adapter imports in default lane).
4. **Re-run the gated E2E** to regenerate `docs/perf_verification_e2e_2026-05-XX.md`
   with the patched assignments and compare against measured.
   - This is the disambiguation moment: if F3 was a wrong assignment, Lecture_2 error
     drops sharply; if not, the error magnitude/sign on Lecture_2 is unchanged.
5. **Conditional on the re-run**:
   - **(5a)** If Lecture_2 fix and any other F1-driven swaps reduce > 2 rooms below
     the 50% error envelope: ship as v0.5 *adapter-data only*; do **NOT** revise
     `MaterialAbsorptionBands` (Finding 4a stays DEFERRED with empirical justification).
   - **(5b)** If errors remain > 50% on > 2 rooms with corrected assignments: D14's
     reverse-trigger fires. Author **ADR 0010 — MaterialAbsorptionBands revision**
     and revise *only* the rows the post-audit error profile justifies (most likely
     `wall_painted`, `wood_floor`, `ceiling_drywall` — the high-area / low-α rows
     Finding 4a flagged). The revision MUST preserve
     `MaterialAbsorptionBands[m][2] == MaterialAbsorption[m]` (band-500 ↔ legacy
     scalar invariant). This implies the legacy `MaterialAbsorption` scalar **also**
     changes for any revised material — which **breaks d-inv-3 byte-equality** of
     `tests/fixtures/golden/sabine_legacy_rt60_500hz.txt`. ADR 0010 must declare this
     and update the golden in the same commit.
6. **Author ADR 0010** (only if 5b fires): "MaterialAbsorptionBands revision per
   F1-confirmed assignments". Required sections: Decision / Drivers / Alternatives
   considered / Why chosen / Consequences / Reverse if / References. Cross-ref D12,
   D13, D14, ADR 0008, ADR 0009.
7. **Author ADR 0011** (always under Scenario A): "ACE Challenge adapter geometry
   table — verified against Eaton 2016 TASLP Table I". Records what verified, the
   diff vs v0.4, and the local Table I artefact path.
8. **Append D15 to `.omc/plans/decisions.md`** summarising:
   - F1 status (DONE — byte-audit complete; cite delta report path).
   - F3 status (DONE if Lecture_2 swapped, or REJECTED if Eaton confirms acoustic
     tile — record either way).
   - F4a status (DONE under 5b, or DEFERRED-with-rationale under 5a).
   - F4b status (independent — pointer to Scenario A's optional concurrent ship; see
     "MISC_SOFT" sub-track below).
9. **Optional concurrent track** (purely additive): ship Finding 4b in the same
   release. See Scenario B steps 1–6 — they run in parallel under Scenario A.
10. **Version bump + release notes + local tag**, identical to v0.4 §7–§9.

### Scenario C — "Predictor-only at v0.5" (Millington-Sette per-surface)

**Trigger**: Reviewer wants v0.5 to ship *something* even if F1 paper is unavailable
**and** F4b is judged premature (e.g., engine team explicitly asks not to extend the
enum yet).

**Goal**: Ship `millington_sette_rt60` and `millington_sette_rt60_per_band` as a third
parallel predictor, mirroring the v0.4 Eyring shape exactly. ADR 0009 already lists this
as an "alternative considered, not for v0.4" — the v0.5 ADR formally promotes it.

**Step-by-step**:

1. **Add `millington_sette_rt60` and `millington_sette_rt60_per_band`** to
   `roomestim/reconstruct/materials.py` (additive only; existing function bodies
   untouched per d-inv-3).
   - Formula: `T = 0.161 V / Σᵢ −Sᵢ ln(1 − αᵢ)` — per-surface decay sum
     (Millington 1932; Vorländer 2020 §4.2).
   - Raise `ValueError` on any `αᵢ ≥ 1.0` per surface (more brittle than Eyring's ᾱ
     average; that's the point — gives a stricter "is the input physically valid"
     check).
   - Raise `ValueError` if `S_total <= 0` (empty surfaces).
   - Raise `ValueError` if every surface has α=0 (denominator zero).
2. **Add 6 unit tests** to `tests/test_room_acoustics_octave.py`:
   - `test_millington_sette_synthetic_shoebox_closed_form`
   - `test_millington_sette_low_absorption_converges_to_sabine`
   - `test_millington_sette_below_eyring_below_sabine_high_alpha`
     (Vorländer ordering: MS ≤ Eyring ≤ Sabine in heavily-absorbed rooms — this is the
     v0.5 monotonicity invariant).
   - `test_millington_sette_raises_on_alpha_one_surface`
   - `test_millington_sette_raises_on_empty_surfaces`
   - `test_millington_sette_per_band_500hz_matches_scalar`
3. **Extend the gated E2E test** to report a third predictor column and assert the
   3-way invariant `millington_sette ≤ eyring ≤ sabine + 1e-9` per room and per band.
   - Row tuples become 8-tuples: `(room_id, sabine, eyring, ms, measured, err_sab,
     err_eyr, err_ms)`.
   - Regenerate `docs/perf_verification_e2e_2026-05-06.md` with three-predictor table.
4. **Author ADR 0010** — "Millington-Sette as third parallel predictor". Cross-ref
   ADR 0009 (which considered this and rejected for v0.4 scope).
5. **Append D15 to `decisions.md`** — "Millington-Sette shipped; F1/F3/F4a/F4b
   re-affirmed as DEFERRED".
6. **Reaffirm F1/F3/F4a/F4b deferrals** in `.omc/plans/v0.5-audit-findings.md` (same
   text as Scenario B step 7, plus a fourth bullet for F4b deferral rationale).
7. **Version bump + release notes + local tag** (same pattern as v0.4 §7–§9).

## §2 Recommended scope (planner's recommendation)

**Recommendation**: ship **Scenario A + Scenario B combined** when the parallel
research report shows Eaton 2016 Table I is acquirable; ship **Scenario B alone** when
it is not. Scenario C is the fallback only if both A and B are blocked (which would
require both an unavailable paper *and* an explicit "do not extend the enum" signal —
unlikely in concert).

Combined Scenario A + B is recommended over A-alone because:

- Scenarios A and B are **structurally orthogonal** (A touches the adapter and the perf
  doc; B touches the model enum and four unit tests). They share only the version
  bump, decisions log entry, and release notes — i.e., the trivial bookkeeping. Risk
  of cross-contamination is low.
- F4b is the only F-finding whose ADR can be written today regardless of paper
  availability. Slipping it past v0.5 forces v0.6 to gain a "schema cleanup" headline,
  which is less useful per release than landing it now.
- Combined scope still respects the SHORT consensus mode (no architecture redesign,
  no Stage-2 schema flip, no production code path beyond `model.py` and
  `adapters/ace_challenge.py`).

If Scenario A's step (5b) fires (coefficient revision needed despite assignment fix),
the resulting commit is materially larger than a SHORT release and **should be
escalated to DELIBERATE consensus mode** with a pre-mortem and expanded test plan
before landing. That decision is human-gated (see §3 OQ-3 and §6 R-V).

Scenario B alone (recommended fallback): self-contained, additive, minimal-risk,
highest schema-cleanliness yield per LoC. Does the right thing if F1 stays blocked.

**Recommended next planner action** (after the parallel research report lands):
re-enter this plan, mark the chosen scenario, and proceed to §4. If Scenario A fires
**and** step 5b is required, escalate to DELIBERATE consensus mode and run the
pre-mortem in §3 OQ-3.

## §3 Open questions for human decision

The questions below cannot be resolved by reading the codebase — they need either an
external artefact (paper) or a human policy call. They are *also* mirrored to
`.omc/plans/open-questions.md` per planner protocol.

- **OQ-1** Eaton 2016 TASLP Table I — is the parallel `cwm:websearchwithme` research
  going to surface a viable acquisition path (legitimate IEEE/ACM, institutional,
  preprint, author email)? The decision between Scenario A and Scenario B hinges on
  this single signal. *Why it matters*: F1 is the upstream gate for F3 and F4a; until
  resolved, three of four DEFERRED items remain DEFERRED.
- **OQ-2** F4b enum coefficients — `MaterialAbsorption[MISC_SOFT] = 0.40` is a
  representative upholstered-furniture mid-band value (Beranek 2004 / Vorländer 2020
  Appx A class). Is the reviewer comfortable shipping this as representative-not-
  verbatim (the same honesty-marker policy that protects the v0.3 `MaterialAbsorptionBands`
  rows), or do we want to wait for a verbatim citation? *Why it matters*: blocks
  Scenario B if "verbatim required" is the policy answer.
- **OQ-3** D14 reverse-trigger semantics — under Scenario A 5b (audit confirms
  assignments are correct AND errors > 50% on > 2 rooms persist), do we (a) revise
  `MaterialAbsorptionBands` in the same v0.5 release with one ADR (ADR 0010)
  documenting both the band table and the legacy scalar regen, or (b) ship the
  audit-only outcome as v0.5.0 and split the coefficient revision into v0.5.1?
  *Why it matters*: option (a) is a larger commit and triggers DELIBERATE consensus
  mode; option (b) splits the story across two tags but each commit is SHORT. Default
  recommendation is (b) — smaller, atomic, easier to review.
- **OQ-4** Stage-2 schema flip (D8) — if F4b lands, do we keep Stage-1 (`additionalProperties: true`)
  or use the v0.5 release as the trigger to flip Stage-2? *Why it matters*: D8 says
  Stage-2 is post-A10-lab-capture human-gated, and lab capture has not happened. The
  default is to **keep Stage-1**; flipping Stage-2 in v0.5 would require A10 to also
  ship. **Default recommendation: keep Stage-1.**
- **OQ-5** ADR 0010 number reservation — under Scenario A combined with B, both new
  ADRs (`misc-soft-enum`, `materials-revision` if 5b fires, `ace-geometry-verified`)
  need numbers. Default ordering: 0010 = MISC_SOFT, 0011 = ACE geometry verified,
  0012 = materials revision (if 5b). *Why it matters*: ADR numbers are immutable
  once committed; pinning the ordering before the release matters for cross-refs.
